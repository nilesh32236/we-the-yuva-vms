## Issues

### Critical (Must Fix)

#### 1. Unbounded in-memory cache — `backend/src/modules/leaderboard/leaderboard.service.ts:8`

The `cache = new Map<string, CacheEntry>()` (line 8) is **never evicted**. Every unique combination of `(scope, timeframe, sortBy, locationId)` adds a permanent entry. While each entry has a 2-minute TTL for staleness (checked on line 23), the Map never calls `.delete()` — so entries accumulate indefinitely under varied query parameters.

**Fix:** Delete expired entries on read: after line 23, add `else { cache.delete(cacheKey); }`. Or use a bounded LRU cache.

#### 2. Missing rate limiter on verify-otp — `backend/src/modules/auth/auth.routes.ts:95`

`POST /auth/verify-otp` (line 95) has **no rate limiter**:
```
authRouter.post('/verify-otp', validate(VerifyOtpSchema), verifyOtpHandler);
```
Both `register` and `send-otp` have rate limiters. Without one on verify-otp, a 6-digit OTP (1,000,000 combinations) can be brute-forced at high throughput. A motivated attacker can exhaust the key space in minutes.

**Fix:** Add `otpLimiter` (or a stricter limiter, e.g., 10/15min) before `validate` on line 95.

#### 3. Broken indentation — `backend/src/workers/notification.worker.ts:52-76`

The `Promise.allSettled` parallel push code (lines 52-76) lost its indentation — it's at indent level 0 instead of inside `sendPushToUser`. This will fail Biome linting and make the code non-functional as written (the `for` loop is outside the `try` block that contains the `const subs` declaration on line 50).

**Fix:** Re-indent lines 52-76 by 4 spaces to be inside the `try` block (after line 50).

#### 4. Broken indentation — `backend/src/workers/notification.worker.ts:885-920`

The batch-fetch code for `check-event-reminders` (lines 885-920) is at indent level 0 instead of inside the `if (job.name === 'check-event-reminders')` block.

**Fix:** Re-indent lines 885-920 to match surrounding nesting.

#### 5. Broken indentation — `backend/src/modules/notifications/notifications.service.ts:58-82`

Same pattern as notification.worker.ts — the `Promise.allSettled` code (lines 58-82) is at indent level 0 inside `sendPushToUser`, where the expected indent is 2 spaces (matching line 56).

**Fix:** Increase indentation by 2 spaces for all lines 58-82.

### Important (Should Fix)

#### 6. Cache entry type loss — `backend/src/modules/leaderboard/leaderboard.service.ts:4`

`CacheEntry.data` is typed as `unknown`. On cache hit (line 24), `cached.data` returns `unknown` to callers, losing all type information. Callers need unsafe casts.

**Fix:** Use generic `CacheEntry<T> { data: T; expiresAt: number }` and type the cache as `Map<string, CacheEntry<...>>`.

#### 7. Unstable cache key — `backend/src/modules/leaderboard/leaderboard.service.ts:12`

`JSON.stringify(params)` for cache keys is order-dependent: `{a:1, b:2}` and `{b:2, a:1}` produce different keys for semantically identical queries.

**Fix:** Use a stable-stringify function (e.g., `json-stable-stringify` or sort keys manually).

#### 8. Stale role ID cache — `backend/src/modules/organizations/organizations.service.ts:8,17`

`coordinatorRoleId` and `volunteerRoleId` are module-level caches that persist for the entire process lifetime. If roles are reseeded (migration, seed script) while the server is running, stale IDs are used silently, potentially assigning wrong roles.

**Fix:** Add TTL (e.g., 5 minutes) or remove caching since role lookups are infrequent and lightweight.

#### 9. Matching pool too small — `backend/src/modules/opportunities/matching.service.ts:83`

Hard `take: 50` on the initial opportunity query limits the candidate pool. If >50 active future opportunities exist, the final top-10 is selected from only the first 50, potentially missing better matches.

**Fix:** Remove the `take` limit or increase it to a larger value (e.g., 500), since the DB query is already scoped to active/future opportunities.

#### 10. Unsafe Error type assertion — `backend/src/modules/badges/badge-engine.service.ts:67`

`(err as Error).message` — caught values can be strings, null, or non-Error objects. `(null as Error).message` would produce `'undefined'` as the logged message.

**Fix:** Use `const message = err instanceof Error ? err.message : String(err)`.

#### 11. deleteNotification swallows all errors — `backend/src/modules/notifications/notifications.service.ts:132-137`

```typescript
try {
  return await prisma.notification.delete({ where: { id: notificationId, userId } });
} catch {
  throw new AppError('Notification not found', 404);
}
```

Transient errors (connection timeout, DB restart) are caught and reported as 404s, misleading the client and masking operational issues.

**Fix:** Check error code: re-throw as 404 only for Prisma `P2025`; re-throw original error for other codes.

#### 12. Promise.allSettled index access — `backend/src/modules/notifications/notifications.service.ts:69-71`

```typescript
for (let i = 0; i < subs.length; i++) {
  if (results[i].status === 'rejected') {
    const reason = (results[i] as PromiseRejectedResult).reason;
```

The `as PromiseRejectedResult` cast bypasses type safety. TypeScript's control flow analysis works properly with `results.forEach()`.

**Fix:** Use `results.forEach((result, i) => { if (result.status === 'rejected') { ... } })`.

*(Same pattern at `notification.worker.ts:61-63` — fix both.)*

#### 13. Redundant error wrapping — `backend/src/modules/upload/upload.service.ts:119-122`

The inner `.catch(() => {})` on line 121 already swallows the `unlink` error. The outer `try { ... } catch {}` is dead code that never executes because `.catch()` returns a resolved promise.

**Fix:** Remove the outer `try-catch` (lines 119-122).

#### 14. `updateLastActive` breaks streak detection — `backend/src/workers/streak.handler.ts:61-67`

`updateLastActive` sets `lastActiveAt: new Date()` (today). If called before the streak handler runs, the handler can't detect yesterday's activity because `lastActiveAt` now points to today, which falls outside the `[yesterday, todayStart)` window.

**Fix:** Either (a) run streak handler before any activity that calls `updateLastActive`, or (b) track "last active date" separately from "last active timestamp" for streak purposes.

#### 15. Missing `.catch()` on queue add — `backend/src/modules/events/events.service.ts:578-584`

```typescript
await notificationsQueue
  .add('attendance-confirmed', { ... })
  .catch(() => {});  // 🔴 silently swallows all errors
```

Other notification queue `.catch()` handlers in this PR log via `logger.warn` (e.g., opportunities.service.ts:61-62). This one silently swallows.

**Fix:** Add `logger.warn` in the catch handler for consistency and debuggability.

### Minor (Nice to Have)

#### 16. `completions: undefined` leaves key — `backend/src/modules/training/training.service.ts:161`

```typescript
return {
  ...l,
  completed: l.completions.length > 0,
  completions: undefined,
};
```

While `JSON.stringify` omits `undefined` values, TypeScript still sees the property as present with `type undefined`. Fragile if serialized differently.

**Fix:** Destructure: `const { completions, ...rest } = l; return { ...rest, completed: completions.length > 0 };`.

#### 17. TOCTOU race on uploads dir — `backend/src/modules/upload/upload.service.ts:11-21`

Two concurrent requests can both see `uploadsDirReady === false` and both call `mkdir`. Harmless (idempotent due to `{ recursive: true }`) but wasteful.

**Fix:** Use a lazy-init promise pattern or `once` wrapper.

#### 18. Silent audit-log failures — `backend/src/modules/admin/admin.service.ts:71-77` (and 3 more)

All four audit `.catch(() => {})` calls silently swallow write failures. While intentional to avoid blocking requests, it hides operational issues.

**Fix:** Log at `warn` level: `.catch((e) => logger.warn('Audit log failed', { error: e.message }))`.

#### 19. Inconsistent fire-and-forget error handling — `backend/src/modules/mentorship/mentorship.service.ts` and others

- `mentorship.service.ts:48` — logs with `logger.warn` on notification failure ✅
- `mentorship.service.ts:124, 194, 202` — silently swallows with `.catch(() => {})` 🔴
- `events.service.ts:584` — silently swallows 🔴

**Fix:** Use consistent `logger.warn` pattern across all fire-and-forget `.catch()` handlers.

#### 20. Badge award race condition — `backend/src/modules/badges/badge-engine.service.ts:34,57`

The read of `earnedBadgeIds` (line 34-40) and the subsequent write of `userBadge.create` (line 53) are not atomic. Two concurrent `checkAndAwardBadges` calls for the same user can both miss the set check and award the same badge.

**Fix:** Wrap the read-check-write cycle in a `$transaction`, or use a unique constraint on `(userId, badgeId)` as a safety net (if not already present).

#### 21. EVENTS_AFTER_HOURS loads all attendances — `backend/src/modules/badges/badge-engine.service.ts:148-153`

Criteria evaluation for `EVENTS_AFTER_HOURS` still calls `attendance.findMany` with no pagination. For power users with thousands of attendances, this loads unnecessary data just to count after-hours events.

**Fix:** Use `attendance.count` with a where filter that checks the event time/day conditions, if the schema supports it.

The missing rate limiter on verify-otp (auth.routes.ts:95) is a security gap enabling OTP brute-force. The remaining important/minor items are real quality concerns but don't block the merge.
