## Changes Made During Review

### 1. Added `aria-describedby` to setup-profile inputs (IMPORTANT)

`frontend/app/(auth)/setup-profile/page.tsx:341` and `frontend/app/(auth)/setup-profile/page.tsx:371`

The register page's inputs use `aria-describedby` pointing to error IDs (e.g., `aria-describedby={errors.name ? 'name-error' : undefined}`), but the setup-profile `commonInput` and `selectInput` helpers didn't link inputs to their errors. Screen readers wouldn't announce field errors on focus.

**Fix**: Added `aria-describedby={error ? `${id}-error` : undefined}` to both `commonInput`'s `<input>` and `selectInput`'s `<select>` elements.

### 2. Added AlertTriangle icon to setup-profile error banner (MINOR)

`frontend/app/(auth)/setup-profile/page.tsx:923`

The setup-profile form error banner was missing the `AlertTriangle` icon that the register page's error banner includes, creating inconsistent UX. Added the icon and the import.

## Remaining Concerns (not fixed)

- **Raw hex colors in error banners** (`frontend/app/(auth)/register/page.tsx:128`, `frontend/app/(auth)/setup-profile/page.tsx:922`) — Uses `bg-red-50`, `text-red-700`, etc. instead of theme tokens like `bg-brand-primary`. The project convention (AGENTS.md) says "always use `bg-brand-primary` etc, never raw hex". However, there are no brand error-themed CSS variables defined (no `bg-brand-error-bg` etc.), so this is a pre-existing gap. Fixing it would require adding new CSS variables, which is outside PR scope.

- **TypeScript `as true` casts** (`frontend/app/(auth)/setup-profile/page.tsx:289-290`) — `privacyPolicyAccepted` and `codeOfConductAccepted` are cast with `as true`. While `validateStep()` guards these at runtime, the type assertion bypasses compile-time safety. Low practical risk since validation runs immediately before submission.
