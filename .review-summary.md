# Review: Multi-step volunteer registration + hide org registration

## Strengths

- **Shared Zod schemas** across frontend/backend (`lib/shared/schemas/onboarding.schemas.ts` and `backend/src/shared/schemas/onboarding.schemas.ts`) — single source of truth for validation
- **Transactional atomicity** in `backend/src/modules/users/users.service.ts:submitOnboarding` — `prisma.$transaction` ensures user update and profile upsert are atomic
- **ChipSelect component** at `frontend/app/(auth)/setup-profile/page.tsx:48` — clean reusable abstraction for multi-select chip toggles
- **VOLUNTEER role guard** in `backend/src/modules/users/users.service.ts` — prevents non-volunteers from submitting onboarding (fix from previous review)
- **Details stored as structured object** `{ steps: {...}, onboardingCompletedAt: ... }` — prevents key collision across steps (fix from previous review)
- **Backward compatibility** — `StaffProfileForm` at line 821 correctly preserves the original flow for non-VOLUNTEER users
- **Register page simplification** — clean removal of role selection with minimal diff
- **Previous review issues (8 items) all addressed** — CORPORATE_CSR dead code, hardcoded consent, missing role guard, bio enum keys, details spread, unused import, aria attributes, DOB validation

## Issues Found

### Important (Should Fix)

| # | File | Issue |
|---|------|-------|
| 1 | `backend/src/shared/schemas/auth.schemas.ts:6` and `frontend/lib/shared/schemas/auth.schemas.ts:6` | **RegisterSchema still allows `ORGANIZATION_ADMIN` role**. The PR removed the role selection UI, but both schemas still permit `role: z.enum(['VOLUNTEER', 'ORGANIZATION_ADMIN'])`. A direct API call can still register as ORGANIZATION_ADMIN. Should restrict to `['VOLUNTEER']` only. |
