## Real Issues

### Critical (Must Fix)

**1. Race condition in `generateEventsFromSeries`**
`backend/src/modules/events/events.service.ts:1008-1075`
- Two concurrent calls to `generateEventsFromSeries` read `series.currentCount` simultaneously, compute identical date ranges, and call `createMany` with overlapping dates. `skipDuplicates: true` prevents duplicate rows (backed by `@@unique([seriesId, eventDate])` at `schema.prisma:480`), but events are silently dropped and `currentCount` is incremented by `count` (the actual inserted count from `prisma.event.createMany`) so the count is correct on the `createMany` result. However, the race window between `findUnique` (line 1012) and `update` (line 1068) means: (a) the same dates could be attempted by both calls, with the second call's events silently dropped, and (b) the second call could generate events beyond `endDate`/`maxOccurrences` because `currentCount` was stale when `calculateNextDates` ran. Recommend using `prisma.$transaction` with `SELECT ... FOR UPDATE` or a distributed lock.

### Important (Should Fix)

**2. ORGANIZATION_ADMIN role missing chat permissions**
`backend/prisma/seed.ts:107`
- `chat:read` and `chat:send` are assigned to VOLUNTEER (lines 38-39), COORDINATOR (lines 71-72), and ADMIN (lines 182-183), but not to ORGANIZATION_ADMIN (line 107). Organization admins cannot participate in opportunity chat rooms.

**3. Raw hex color classes in volunteer events page**
`frontend/app/(volunteer)/volunteer/events/page.tsx:25-28`
- `STATUS_COLORS` uses `bg-blue-100`/`bg-red-100`/`bg-purple-100` for status badges. These are raw Tailwind color classes, not theme tokens. While AGENTS.md says "always use `bg-brand-primary` etc, never raw hex", these are Tailwind utility classes (not hex values) and the AGENTS.md convention is about CSS variable tokens. Still, using `bg-blue-100` breaks dark mode consistency. The `EventCard.tsx` at lines 7-11 shows the correct pattern using `bg-brand-cta/10`, `bg-brand-primary/10`, `bg-brand-error/10`.

**4. BadgeApprovalQueue uses `request.badge.id` instead of request id**
`frontend/components/badges/BadgeApprovalQueue.tsx:39-41`
- The API URL is constructed as `/badges/${request.user.id}/${request.badge.id}/approve` which uses the user and badge IDs from the pending approval's related records. The backend route at `badges.routes.ts:11` expects `/:userId/:badgeId/approve`. This is functionally correct since `listPendingApprovals()` returns `include: { user, badge }` relations. However, the route structure with user/badge IDs in the URL path means the backend must trust these client-provided IDs. In an approval workflow, the approval record's own `id` would be a safer parameter. As-is, anyone with `BADGE_APPROVE` could construct arbitrary approve/reject requests by passing different userId/badgeId combinations, though this is partially mitigated by RBAC.

**5. Unpaginated CSV export**
`backend/src/modules/events/events.service.ts:1085-1094`
- `exportEventsCsv` loads ALL events matching the filter into memory with no limit. For large organizations with thousands of events, this will consume excessive memory. Recommend streaming or paginated iteration.

**6. `getCoordinatorVolunteers` export also unpaginated**
`backend/src/modules/users/users.service.ts:132-162`
- Same pattern as CSV export — loads all matching volunteers into memory. Fine for small orgs, but will OOM at scale.
